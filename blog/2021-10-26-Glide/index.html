Glide

Glide简介

  • Glide是一个快速高效的Android图片加载库,注重于平滑的滚动。Glide提供了易用的API,高性能、可扩展的图片解码管道(decode pipeline) ,以及自动的资源池技术。
  • Glide支持拉取,解码和展示视频快照,图片,和GIF动画。
  • Glide的Api是如此的灵活,开发者甚至可以插入和替换成自己喜爱的任何网络栈。默认情况下, Glide使用的是一个定制化的基于HttpUrlConnection的栈,但同时也提供了与Google Volley和Square OkHttp快速集成的工具库。

使用

Generated API

  • Generated API默认名为GlideApp,与Application模块中AppGlideModule的子类包名相同。在Application模块中将Glide.with)替换为GlideApp.with),即可使用该API去完成加载工作。使用GlideExtension可以拓展该Api并且实现自定义的功能。

  • GlideExtension: @GlideExtension注解用于标识一个扩展Glide API的类。

  • GlideOption:用@GlideOption注解的静态方法用于扩展RequestOptions
  • GlideType:被@GlideType注解的静态方法用于扩展RequestManager,被@GlideType注解的方法允许你添加对新的资源类型的支持,包括指定默认选项。 用于控制RequestBuilder

通过 LinkedHashMap实现LRUCache最近最少原则

  • 存在一个LinkedHashMap存放数据,并且实现了LRU (最少使用算法)缓存策略。

  • Мap<T, Y> cache = new LinkedHashMap<(100, 0.75f, true):

    • 其中第二个參数0.75表示加载因子,即容量达到75%的时候会把内存临时增加一倍。
    • 最后这个参数也至关重要,表示访问元素的排序方式, true表示按照访问顺序排序, false表示按照插入的顺序排序。
    • LruCache实现原理:·利用了LinkedHashMap排序方式的特性:由于使用访问顺序排序,进行了get/put操作的元素会放在Map最后面。所以当最后一个元素插入进来时,如果当前的缓存数据大小超过了最大限制那么会删除Map中放在前面的元素。

​ 如图，当我们使用put方法存放数据是会将最新的数据放到最下面

当我们通过get()方法获取A时会将A重新排序。放到底部

当集合内存需要恢复到原来大小时会将排在最前面的缓存移除，直到恢复到原来的大小

磁盘缓存

磁盘缓存,支持以下几种策略的选择:

  • DiskCacheStrategy.NONE:表示不缓存任何内容
  • DiskCachestrategy.DATA: 表示只缓存原始图片
  • DiskCacheStrategy.RESOURCE: 表示只缓存转换过后的图片
  • DiskCacheStrategy.ALL: 表示既缓存原始图片,也缓存转换过后的图片(默认选择)
  • DiskCacheStrategy.AUTOMATIC:表示让Glide根据图片资源智能地选择使用哪一种缓存策略(默认选项
